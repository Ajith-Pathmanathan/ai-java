# Module 05: LangChain4j + Agentic Workflows — Research Questions

**Q1: What is the `@AiService` pattern in LangChain4j? How does it compare to Spring AI's `ChatClient`? What are the pros and cons of each?**
*Reason: @AiService: annotate an interface → LangChain4j generates an implementation at runtime. Pros: clean API, no boilerplate, type-safe. Cons: magic (harder to understand/debug), less flexible for dynamic prompts. Spring AI ChatClient: explicit builder pattern — you build the prompt in code. Pros: explicit, debuggable, flexible. Cons: more verbose. Rule: use @AiService for stable AI interactions (fixed system prompts, structured output). Use ChatClient for dynamic interactions where you build prompts at runtime. In practice, @AiService is preferred for services, ChatClient for controllers with user-specific context.*

**Q2: What is `ChatMemoryStore` in LangChain4j? Implement a PostgreSQL-backed ChatMemoryStore for a payment support chatbot that retains conversation history across server restarts.**
*Reason: ChatMemoryStore: interface with three methods — getMessages(memoryId), updateMessages(memoryId, messages), deleteMessages(memoryId). PostgreSQL implementation: table `chat_memory(session_id, message_type, content, created_at)`. JDBC implementation: getMessages → SELECT ordered by created_at; updateMessages → DELETE all + INSERT new. This is production-ready persistent memory. Without this, every server restart forgets all conversations. In fintech: this also serves as an audit trail of every customer-AI interaction.*

**Q3: What is the difference between MessageWindowChatMemory and summarizing memory? When would you use each for a customer support bot?**
*Reason: MessageWindowChatMemory: keeps the last N messages. Fast, predictable, but loses context from early in the conversation. Good for: short focused support sessions. Summarizing memory: when memory is full, summarize old messages with an LLM, replace them with the summary. Preserves conceptual context while controlling token count. Good for: long-running conversations, escalations that span multiple sessions. Implementation: count messages → when > threshold → call LLM to summarize oldest half → replace with SystemMessage("Summary: ..."). The cost: one extra LLM call when memory fills up. Worth it for conversations that span more than 20 messages.*

**Q4: A customer support agent needs to handle this 3-step workflow: (1) extract payment ID from message, (2) look up payment in database, (3) generate empathetic response. How would you implement this as a LangChain4j agent chain vs three separate LangChain4j AI Services? What are the trade-offs?**
*Reason: Chain: one agent does all 3 steps automatically (calls tools in sequence, agent decides the flow). Pros: flexible, agent can adapt if step 2 fails. Cons: harder to test, less predictable, expensive (many LLM calls). Three AI Services: ExtractionService → ToolService → ResponseService (orchestrated by Java code). Pros: predictable, testable, each service has clear responsibility, each can have different models (cheap extraction model + expensive response model). Cons: rigid pipeline, doesn't adapt to unexpected situations. Rule: use explicit service chains for well-defined workflows. Use an agent for workflows where the path is uncertain or depends on intermediate results.*

**Q5: Your LangChain4j agent starts hallucinating tool call parameters — calling `getPayment("PAY-UNKNOWN")` when the user mentioned "my last payment." How do you fix this?**
*Reason: Root cause: ambiguous tool description or the agent doesn't know how to look up "last payment" without a payment ID. Fixes: (1) Add a `getLastPayment(customerId)` tool — give the agent the right tool for the task. (2) Improve tool description: "If the user says 'my last payment' or 'my recent payment', use search_customer_payments first to get the payment ID, then use get_payment." (3) Add a validation layer in the tool: if paymentId doesn't match the format PAY-XXXXX, return an error with the correct format. Key insight: hallucination in tool parameters is almost always a tool description problem, not a model problem.*

**Q6: What is the difference between tool use in LangChain4j and RAG in LangChain4j? When would you combine both in a single agent?**
*Reason: RAG: retrieve relevant documents from a vector store and inject them into the prompt (read-only, no actions). Tool use: agent calls functions that can read from or write to systems (databases, APIs, email, etc.). Combining both: agent first checks policy documents via RAG (what should we do?), then uses tools to take action (actually do it). Example: customer asks for refund → RAG checks refund policy → agent decides if eligible → calls initiate_refund tool → sends confirmation email via email tool. The agent needs BOTH: policy knowledge (RAG) and action capability (tools). In LangChain4j: `AiServices.builder().contentRetriever(ragRetriever).tools(toolsObject).build()`.*

**Q7: You're building an agentic workflow for dispute resolution that spans 3 days (customer submits dispute → merchant has 48h to respond → agent reviews and decides). How do you architect this with LangChain4j and Spring?**
*Reason: This is an agentic workflow, not a simple agent. Requirements: async, stateful, long-running. Architecture: (1) DisputeEntity in PostgreSQL with status field (SUBMITTED, WAITING_MERCHANT, UNDER_REVIEW, RESOLVED). (2) Spring @Scheduled job checks for disputes where merchant deadline has passed. (3) LangChain4j agent reviews the dispute + merchant response + policy → recommends decision. (4) Human-in-the-loop: for high-value disputes, agent recommends but human approves. LangChain4j provides the AI reasoning; Spring provides the workflow orchestration. Key: AI does the analysis, Spring + database manage the state machine.*

**Q8: What is streaming in LangChain4j (TokenStream)? Implement a Spring WebFlux SSE endpoint that streams agent responses token-by-token.**
*Reason: TokenStream: LangChain4j's streaming API. Unlike Spring AI's Flux<String>, TokenStream uses callbacks: onNext(token), onComplete(response), onError(error). To convert to Flux<String>: use Flux.create and bridge the callbacks to the Flux sink. SSE endpoint: produces `text/event-stream`, returns Flux<String>. The agent calls tools normally (tools are not streamed) but the final text response streams token by token. Why important: without streaming, the customer sees nothing for 5+ seconds while the agent calls tools and generates a long response. With streaming, they see the response appear as it's generated.*

**Q9: How do you test a LangChain4j AI Service without calling the real LLM API in every unit test?**
*Reason: Options: (1) Mock the ChatLanguageModel with Mockito — return a canned response for specific inputs. (2) Use a deterministic model stub: implement ChatLanguageModel with fixed responses based on input patterns. (3) Integration tests with a cheap local model (Ollama llama3.1 is free). (4) Record/replay: record real LLM responses once, replay them in tests (similar to VCR for HTTP). Best practice: unit tests mock the LLM, integration tests use a real (local) model. For testing tool calls: verify that the LLM was called with the right system prompt, and the tools were called with the right parameters. Never run LLM integration tests in CI without a cost budget.*

**Q10: Compare the memory management approaches: MessageWindowChatMemory, TokenWindowChatMemory, and entity memory. Implement entity memory for a payment chatbot that must always remember the payment ID even in a 50-message conversation.**
*Reason: MessageWindowChatMemory: last N messages (simple). TokenWindowChatMemory: last N tokens (context-aware). Entity memory: extract key entities and always include them. Implementation: `EntityMemory` that after every message, runs an LLM to extract payment IDs mentioned → stores them in a separate "entity store" → always prepends "Known payment IDs: PAY-123, PAY-456" to the system prompt regardless of which messages are dropped. Why matters: in a 50-message conversation, MessageWindowChatMemory might drop the message where the customer mentioned "PAY-123" — agent loses track. Entity memory ensures critical facts survive context window compression.*
