# Module 03: RAG Systems with Spring AI — Research Questions

**Q1: A product manager says: "We want our chatbot to answer questions using our internal payment policy PDFs." Walk through every component you would build, in order, with technology choices for each.**
*Reason: This is the single most common AI architect interview question in 2025. Answer: (1) Ingestion pipeline — PagePdfDocumentReader → TokenTextSplitter (400 tokens, 80 overlap) → text-embedding-3-small → pgvector. (2) Query pipeline — embed user query → similarity search (topK=5, threshold=0.65) → inject chunks into prompt → GPT-4o generates grounded answer. Components: Spring Boot service, pgvector (or Qdrant), Spring AI QuestionAnswerAdvisor, PostgreSQL. If you can walk through this fluently in 3 minutes, you are hireable for senior AI roles.*

**Q2: What is the difference between naive RAG and advanced RAG? Your naive RAG system gives wrong answers 30% of the time. What are three specific techniques you would add to improve it?**
*Reason: Naive RAG = embed query → search → stuff into prompt → generate. Advanced RAG adds: query transformation (HyDE, query rewriting), hybrid search (keyword + semantic), re-ranking (LLM or Cohere), metadata filtering, and sentence-window retrieval. Three techniques to add: (1) HyDE — generate a hypothetical answer, embed it, search with that. (2) Re-ranking — have an LLM score each retrieved chunk's relevance before using it. (3) Hybrid search — combine keyword (BM25) with semantic search using RRF. Each typically adds 5-15% accuracy improvement.*

**Q3: What is HyDE (Hypothetical Document Embedding)? Why does it work? Write the Spring AI code to implement it.**
*Reason: HyDE: generate a hypothetical answer to the question → embed that answer → use it for retrieval. It works because: questions and answers have different embeddings (questions are sparse, answers are detailed). The hypothetical answer's embedding is in the "answer space" and much closer to real document chunks than the question embedding. Result: 10-20% better retrieval recall. Code: call chatClient to generate hypothetical answer → call embeddingModel.embed(hypotheticalAnswer) → use that vector for vectorStore.similaritySearch.*

**Q4: What is Reciprocal Rank Fusion (RRF)? How do you use it to combine keyword search results and semantic search results?**
*Reason: RRF is the standard way to merge results from different search systems. Formula: score(d) = Σ 1/(rank_i(d) + k) where k=60 is the constant, rank_i is the document's rank in list i. If a document appears in position 1 in keyword results AND position 2 in semantic results, it scores higher than one that only appears in one list. RRF handles the "different scales" problem — you can't just add cosine scores and BM25 scores because they're on different scales. RRF works on ranks, not scores. This is used in all production hybrid search systems.*

**Q5: Your RAG system is confidently giving wrong answers — what you call hallucination. How do you detect and reduce it?**
*Reason: Detection: (1) Faithfulness check — after generating, ask an LLM "Is every claim in this answer supported by the provided context? Answer YES/NO + list any unsupported claims." (2) User feedback — thumbs up/down on every answer. (3) Ground truth evaluation with RAGAS. Reduction: (1) Stronger system prompt — "ONLY answer from context. If not in context, say 'I don't have that information'." (2) Temperature=0.0 (less creative = less hallucination). (3) Raise retrieval threshold (only use high-confidence chunks). (4) Reduce topK (fewer chunks = less noise). An architect who only reduces hallucination reactively is behind; build detection in from day one.*

**Q6: How do you handle conversational RAG where the user asks follow-up questions that reference earlier context? For example: "What is the refund policy?" → "How long does THAT take?"**
*Reason: "THAT" refers to the refund process from the previous answer — no way to search for it in isolation. Solution: query reformulation. Before retrieval, send the current message + conversation history to an LLM with prompt "Rewrite this message as a standalone search query given the conversation history." The reformulated query is then used for vector search. This is called "contextual compression" or "query reformulation". Without it, conversational RAG breaks on the first follow-up question.*

**Q7: What is re-ranking in RAG? When is it worth the extra latency and cost?**
*Reason: Initial retrieval (topK=20) has high recall but low precision — it returns many potentially relevant docs. Re-ranking (topK → top5 from those 20): use a more expensive model (cross-encoder, LLM-as-judge, or Cohere Rerank) to score each candidate for actual relevance. Worth it when: (1) Initial retrieval is noisy (many irrelevant chunks). (2) User queries are complex or ambiguous. (3) Accuracy > latency (policy Q&A, medical advice). Not worth it when: (1) Latency is critical (<200ms requirement). (2) Initial retrieval is already precise (well-structured, filtered documents). A typical production setup: retrieve top-20, rerank to top-5.*

**Q8: Design the database schema for a multi-tenant RAG system where Company A should never see Company B's documents. What changes are needed in both the ingestion pipeline and the query pipeline?**
*Reason: Multi-tenancy in RAG is a security requirement. Schema: add `tenant_id` column to document_embeddings table. Add an index on `metadata->>'tenant_id'`. Ingestion: always include `tenant_id` in document metadata. Query: ALWAYS add `WHERE metadata->>'tenant_id' = :tenantId` BEFORE the vector search. In Spring AI filter expressions: `"tenant_id == '" + tenantId + "'"`. Never allow cross-tenant search. Audit log every query. A missing tenant filter is a critical security bug — one test case must verify this.*

**Q9: A RAG system works well for English documents but poorly for documents in Portuguese. What are the possible causes and fixes?**
*Reason: Multilingual RAG issues: (1) Embedding model: text-embedding-3-small has decent multilingual support, but Cohere Embed v3 and multilingual-e5-large are better for non-English. (2) Chunking: sentence boundaries differ by language — Portuguese needs different sentence detection. (3) Query language mismatch: user asks in Portuguese, document is Portuguese, but the chunking lost structure. Fix: use multilingual embedding models, chunk at paragraph boundaries, test with native language queries. For production multilingual: use multilingual-e5-large or OpenAI text-embedding-3-large (better multilingual). Always embed in the same language as the query.*

**Q10: What is RAGAS? How do you use it to evaluate RAG system quality? What are the four key metrics?**
*Reason: RAGAS (RAG Assessment) is the standard framework for evaluating RAG pipelines. Four metrics: (1) Faithfulness — does the answer only contain claims supported by the context? (2) Answer relevance — does the answer address the question? (3) Context precision — are the retrieved chunks actually relevant to the question? (4) Context recall — did retrieval get all the relevant chunks? You provide: question + ground truth answer + retrieved contexts + generated answer. RAGAS uses an LLM to score each metric 0-1. A production RAG system should have all four metrics above 0.8. Run RAGAS on a test set of 50-100 question-answer pairs before shipping.*
