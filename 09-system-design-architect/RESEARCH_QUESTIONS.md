# Module 09: System Design for Architect Level — Research Questions

**Q1: Design a global payment processing system like Stripe. Handle 10M transactions/day, 99.999% uptime, PCI DSS compliance, and idempotency. Walk through every component.**
*Reason: This is the senior architect interview question for fintech. Key components: (1) API Gateway with idempotency key validation (store in Redis, 24h TTL). (2) Payment Service: validates, saves to PostgreSQL (source of truth), publishes to Kafka. (3) Processor Integration Layer: connects to Visa/Mastercard/bank APIs with timeout + retry. (4) Reconciliation Service: nightly job comparing our records vs bank records (catches silent failures). (5) Multi-region: active-passive (US-East primary, EU-West DR). PostgreSQL with logical replication across regions. (6) PCI DSS: card data in separate PCI-scoped segment, HSM for encryption, no card data in logs or LLM prompts. Answer should include: data flow, error handling, the dual-write pattern for cross-region consistency, and idempotency implementation. This question takes 45 minutes to answer well.*

**Q2: Walk through the 6-step system design framework for "Design a knowledge base Q&A system using RAG for a company with 5M internal documents." Include capacity estimation.**
*Reason: Framework: (1) Functional: semantic search, Q&A with sources, admin can add/remove documents. (2) Non-functional: 10K users, < 3s P95, 99.9% uptime. (3) Capacity: 5M docs × 500 tokens/chunk × 1536 dims × 4 bytes = 15GB embeddings. 10K users × 30 queries/day = 300K/day = 3.5 QPS avg, 35 QPS peak. Cost: 35 QPS × 3000 tokens × $0.005/1K = $0.52/second = $45,000/month → must cache. (4) High-level: ingestion pipeline (S3 → Kafka → workers) + query pipeline (cache → Qdrant → GPT-4o). (5) Detailed: schema, tenant isolation, chunking strategy. (6) Bottlenecks: LLM latency, LLM cost (must cache), ingestion throughput. AI: the entire system IS the AI integration. This framework gives a structured answer in 45 minutes.*

**Q3: What is consistent hashing? How is it used in distributed caches (Redis Cluster) and why is it relevant to scaling vector databases?**
*Reason: Consistent hashing: place nodes and keys on a ring. Key goes to the nearest node clockwise. When a node is added/removed, only keys on that arc are redistributed (O(K/N) remapping vs O(K) for standard hashing). Redis Cluster uses hash slots (16384) distributed across nodes — a form of consistent hashing. For vector databases: Qdrant uses consistent hashing to distribute vectors across shards. When adding a shard, only vectors that hash to the new shard are moved. For an architect: when you design a distributed cache or vector store, you must understand that adding/removing nodes triggers data movement — plan maintenance windows accordingly. Qdrant's distributed mode uses this to shard 1B+ vectors across multiple nodes.*

**Q4: You're asked to design a real-time fraud detection system. Explain how you would combine rule-based, ML model, and LLM-based detection in a layered architecture that meets a 200ms decision deadline.**
*Reason: Three-layer design: (1) Rule-based (< 5ms): Redis lookup — is card on blocklist? More than 5 transactions in 60 seconds? These are deterministic, fast, catch obvious fraud. (2) ML model (< 50ms): XGBoost/LightGBM in-memory model, pre-trained, no network call. Features: amount, merchant category, hour of day, country mismatch, etc. Score 0-100. (3) LLM analysis (< 150ms): ONLY for borderline score 40-60. Lightweight model (GPT-3.5, not GPT-4o) with few-shot examples. Agent queries customer history + merchant reputation. The architecture insight: LLMs are too slow for every transaction at scale. Use them as a "second opinion" for uncertain cases only. This design handles 10M transactions/day with < 5% requiring LLM analysis.*

**Q5: What is a data lake vs a data warehouse? How do you design the data architecture for an AI system that needs both real-time payment data (for fraud detection) and historical analytics (for trend analysis)?**
*Reason: Data lake: raw data in S3/GCS, all formats, schema-on-read. Cheap storage. Good for: ML training data, raw logs, data scientists exploring. Data warehouse: structured, schema-on-write, optimized for analytics queries. BigQuery/Redshift/Snowflake. Good for: BI dashboards, SQL analytics. Lambda architecture for payments: (1) Speed layer: Kafka + ClickHouse → real-time aggregations for fraud detection (last 60 minutes of transaction patterns). (2) Batch layer: Spark jobs → S3 data lake → Redshift warehouse → monthly reports, ML training. (3) Serving layer: combines both. For AI: training data from the data lake (months of historical data), inference features from ClickHouse (last 60-minute window). Both are needed: historical patterns train the model, real-time data feeds the inference.*

**Q6: Design the database schema for a multi-tenant RAG system supporting 1000 companies, each with up to 100K documents. What indexing strategy do you use for security isolation and performance?**
*Reason: Schema: documents(id UUID, tenant_id UUID, filename, content_hash, created_at), document_chunks(id UUID, document_id, tenant_id, content, embedding vector(1536), metadata JSONB, created_at). Indexes: (1) `USING hnsw (embedding vector_cosine_ops)` — for fast similarity search. (2) `(tenant_id, created_at)` — for tenant document listing. (3) GIN on metadata — for filtered search. Security: EVERY query includes `WHERE tenant_id = :tenantId`. Test: write an integration test that inserts documents for tenant A, queries as tenant B, and verifies 0 results. Row-Level Security (PostgreSQL): `ALTER TABLE document_chunks ENABLE ROW LEVEL SECURITY; CREATE POLICY tenant_isolation ON document_chunks USING (tenant_id = current_setting('app.tenant_id')::UUID)`. This makes it impossible to retrieve another tenant's data even if there's a bug in the application.*

**Q7: You need to design a system that ingests 1 million documents per day and makes them searchable within 5 minutes of upload. Design the ingestion pipeline architecture.**
*Reason: 1M docs/day = 11.6 docs/second average. With text extraction + embedding + storage = ~30 seconds per doc (bottleneck: embedding API). At 11.6 docs/sec, need: 11.6 × 30 = 348 concurrent processing slots. Architecture: (1) Document upload → S3 + publish event to Kafka (fast, decoupled). (2) Kafka topic: `document-uploaded`, 50 partitions. (3) Consumer pods: 50 pods × 6 concurrent workers = 300 concurrent document processors. (4) Each worker: download from S3 → text extraction (Tika) → chunking → batch embedding (50 chunks per OpenAI call) → Qdrant upsert. (5) State: processing_status table (PENDING, PROCESSING, COMPLETE, FAILED). (6) 5-minute SLA: monitor Kafka consumer lag. Alert if lag grows (ingestion falling behind). Scale up consumer pods if needed (KEDA → add pods when lag > 5000 messages).*

**Q8: Explain the write-ahead log (WAL) in PostgreSQL and how it enables Change Data Capture (CDC) with Debezium. Design a pipeline using CDC to keep a vector database in sync with a source database.**
*Reason: WAL: every change (INSERT/UPDATE/DELETE) in PostgreSQL is first written to the WAL (a sequential log file) before the actual data file is modified. This guarantees durability. Debezium reads the WAL via PostgreSQL logical replication → publishes change events to Kafka. Pipeline: PostgreSQL (source) → Debezium → Kafka (db.public.documents) → Embedding Service → Qdrant (upsert or delete). Events: INSERT → embed and upsert to Qdrant. UPDATE → re-embed and upsert. DELETE → delete from Qdrant by document_id. This keeps the vector DB in sync without polling. Latency: changes appear in Qdrant within 1-5 seconds of the DB write. For RAG systems: documents added to PostgreSQL automatically appear in vector search within seconds. No manual re-ingestion needed.*

**Q9: A product manager asks for AI-powered features on your payment platform. How do you decide which features to build with AI vs without? Give a decision framework.**
*Reason: Decision framework — ask these questions: (1) Is the task well-defined enough for traditional rules? (fraud blocklist for known bad cards → rules, not AI). (2) Is the answer probabilistic/contextual? (is this transaction unusual for this customer? → AI). (3) Does it require understanding language/semantics? (extract payment intent from a customer email → AI). (4) What is the accuracy requirement? (block a payment based on AI → must be > 99.9% accurate → human-in-the-loop + AI). (5) What is the cost? (LLM costs money per call → only use AI where it clearly beats the alternative). (6) What is the latency requirement? (< 200ms → no LLM, use ML model). AI is the right choice when: (a) language understanding is needed, (b) the task is too complex for hand-written rules, (c) you have data to validate it. Use AI as a last resort, not a first instinct.*

**Q10: What is an Architecture Decision Record (ADR)? Write an ADR for choosing PostgreSQL pgvector over Qdrant for a startup with an existing PostgreSQL deployment.**
*Reason: ADR: a short document capturing an architectural decision, its context, and alternatives considered. ADR template: Title, Status, Context, Decision, Consequences, Alternatives. Example: Title: ADR-007: Use pgvector for document search. Context: We need semantic search for 500K documents. We already run PostgreSQL (RDS). We have 3 engineers, no dedicated infrastructure team. Decision: Use pgvector. Consequences: Positive: no new service to manage, ACID transactions span our data and embeddings, team already knows PostgreSQL. Negative: lower throughput than Qdrant at > 5M vectors, no HNSW until pgvector 0.5+. Risks: if we grow to 10M vectors, we may need to migrate to Qdrant. Alternatives: Qdrant — better performance but adds a new service. Weaviate — too complex for our team size. Chroma — not production-ready for our scale. ADRs prevent the "why did we choose X?" question from being answered by "I don't know, it was before my time."*
