# Module 02: Embeddings + Vector Databases — Research Questions

**Q1: What is an embedding? If you embed the sentence "Payment declined" using text-embedding-3-small, what do you get back? Why can't you reverse the embedding back to text?**
*Reason: Embeddings are the foundation of all semantic AI systems. text-embedding-3-small returns a float[1536] — 1536 numbers representing the meaning of the text in a high-dimensional space. They are one-way: many different texts can have similar embeddings (similar meaning), so you cannot reconstruct the original text. An architect who doesn't understand this builds systems that try to "decode" embeddings — a common junior mistake.*

**Q2: What is cosine similarity? Write the formula. Why do we use cosine similarity instead of Euclidean distance for text embeddings?**
*Reason: Cosine similarity = dot(A,B) / (|A| × |B|). It measures the angle between vectors, not the distance. Text embeddings for longer texts have larger magnitudes (longer vectors) even if the meaning is identical to a shorter text. Euclidean distance would rank "long text about payments" as MORE different from "short text about payments" than two completely unrelated texts. Cosine similarity normalizes for magnitude — so meaning is what matters, not text length. This is the standard interview question for vector DB positions.*

**Q3: You have 1 million support ticket records. You want to find tickets semantically similar to "customer charged twice for same transaction". Walk through the exact steps from raw text to returning the top 5 results. Include all components.**
*Reason: This is the architect-level question. Answer: (1) Embed query using text-embedding-3-small → float[1536]. (2) Run ANN (approximate nearest neighbour) search in pgvector/Qdrant using HNSW index. (3) Use cosine distance operator `<=>` with `ORDER BY ... LIMIT 5`. (4) Return the 5 closest tickets by cosine distance. Understanding each step — embedding, indexing, ANN vs exact search — shows you understand the full pipeline.*

**Q4: What is the difference between IVFFlat and HNSW indexes in pgvector? When would you choose each?**
*Reason: IVFFlat: clusters vectors into "lists", searches only the nearest clusters. Faster to build, lower memory. Best for: large datasets where you want to tune recall vs speed. HNSW: Hierarchical Navigable Small World graph. Higher memory, faster queries at query time, better recall. Best for: production systems where query latency matters. The rule: use HNSW for production (pgvector 0.5+). IVFFlat requires you to set `lists` correctly (≈ √row_count). HNSW is more forgiving. An architect chooses the right index based on data size and query volume.*

**Q5: Your chunking strategy is producing poor semantic search results. You're using 1000-token chunks with no overlap. What problems does this cause and how do you fix it?**
*Reason: 1000-token chunks are too large — the embedding averages the meaning of too much text, diluting the specific information that would match a query. No overlap means sentences at chunk boundaries lose their context (the beginning of a chunk doesn't know the ending of the previous chunk). Fix: reduce to 300-500 tokens, add 50-100 token overlap. Good overlap: 15-20% of chunk size. This question separates engineers who copy-paste vector DB code from those who understand retrieval quality.*

**Q6: What is metadata filtering in a vector database? You have 500K embedded documents across 20 document types. A user queries "payment refund policy" but should only see documents from their company (tenant_id=42). Write the SQL/filter expression.**
*Reason: Pure semantic search across all documents is dangerous in multi-tenant systems — a user could semantically match documents from another company. Metadata filtering: store tenant_id in JSONB metadata, add a WHERE clause before the vector search. In pgvector: `WHERE metadata->>'tenant_id' = '42' ORDER BY embedding <=> :queryVec LIMIT 10`. In Qdrant: payload filter `{"must": [{"key": "tenant_id", "match": {"value": 42}}]}`. This is a security requirement, not a nice-to-have.*

**Q7: When would you use Qdrant instead of pgvector? A startup is building a new AI-powered document search system with no existing database. They expect 50M documents and 10,000 searches per second at peak. What is your recommendation and why?**
*Reason: At 50M vectors and 10K QPS, pgvector reaches its limits. Qdrant is Rust-based, purpose-built for vector search, handles this scale with quantization (scalar/product quantization reduces memory 4-8x). Recommendation: Qdrant — it's Docker-ready, gRPC for high throughput, payload indexes for fast filtering, and handles 10K QPS on commodity hardware. pgvector is the right choice when: you already run PostgreSQL, you need ACID transactions across business data AND vectors, and scale is <5M vectors with <1K QPS.*

**Q8: What is semantic chunking and when is it worth the extra cost? Compare it to fixed-size and sentence-based chunking for a 200-page payment policy document.**
*Reason: Semantic chunking uses an LLM to identify natural topic boundaries — every chunk is a coherent topic, not an arbitrary N-token slice. This significantly improves retrieval quality because queries match chunks on topic, not fragmented ideas. Cost: requires one extra LLM call per document section. For a 200-page policy document: fixed-size = fast, cheap, mediocre quality. Sentence-based = better quality, easy to implement. Semantic = best quality, 10-50x more expensive in LLM calls. Rule: semantic chunking is worth it for high-value documents that will be queried millions of times.*

**Q9: Your similarity search returns irrelevant results even though the similarity scores are above 0.7. What are the possible causes and how do you diagnose each?**
*Reason: High score ≠ relevance. Causes: (1) The query embedding is too generic — the query "payment" matches everything about payments equally. Fix: make queries more specific. (2) Chunks are too large — the embedding averages a lot of meaning. Fix: reduce chunk size. (3) Wrong similarity threshold for your model — 0.7 means different things for different embedding models. Fix: calibrate thresholds empirically. (4) The embedded content doesn't actually contain the answer — RAG retrieval failure. Fix: inspect the actual chunks being returned. This diagnostic thinking separates architects from coders.*

**Q10: You are building a fintech system where payment transaction data cannot leave your infrastructure (PCI DSS). You need semantic search over 10M transaction descriptions. What is your complete architecture: embedding model, vector database, deployment?**
*Reason: PCI DSS: no data to external APIs. Architecture: (1) Embedding: nomic-embed-text or mxbai-embed-large via Ollama — runs locally on your infra, data never leaves. OR Azure OpenAI with a data processing agreement. (2) Vector DB: pgvector if you already run PostgreSQL; Qdrant deployed in your Kubernetes cluster if starting fresh. (3) Deployment: Docker container with Ollama model downloaded at build time, volume-mounted for persistence. This question is asked in every fintech AI architect interview.*
