# Module 08: Kubernetes + Cloud Architecture — Research Questions

**Q1: A Spring Boot AI service is running in Kubernetes and crashing randomly due to OutOfMemoryError. How do you diagnose and fix it?**
*Reason: Diagnosis: (1) kubectl logs <pod> → look for OOM messages. (2) kubectl describe pod <pod> → look for "OOMKilled" in LastState. (3) kubectl top pod → check memory usage trend. Root causes: (1) Missing -XX:+UseContainerSupport (JVM doesn't know it's in a container, uses host RAM). (2) Resource limits.memory too low. (3) Memory leak in the application. (4) LLM responses being accumulated without bounds (long conversation history). Fix: add JVM flags: `-XX:+UseContainerSupport -XX:MaxRAMPercentage=75`. Set resource requests and limits. Add jmap heap dump in preStop hook if you need more detail. For AI services: bound conversation history length, use streaming instead of accumulating full responses.*

**Q2: How do you pass an OpenAI API key to a Spring Boot pod in Kubernetes securely? Walk through the complete solution from secret creation to pod consumption.**
*Reason: Never hardcode API keys. Never put them in ConfigMaps (not encrypted). Solution: (1) Create Kubernetes Secret: `kubectl create secret generic ai-secrets --from-literal=openai-api-key=sk-... -n fintech`. (2) Mount as environment variable in deployment.yaml: `env: - name: OPENAI_API_KEY valueFrom: secretKeyRef: name: ai-secrets key: openai-api-key`. (3) Spring Boot reads it: `spring.ai.openai.api-key=${OPENAI_API_KEY}`. For production: use External Secrets Operator + AWS Secrets Manager. Secret is stored in AWS, synced to Kubernetes Secret automatically. Rotation: update in AWS → External Secrets rotates the K8s Secret → pod restart picks up new key. Never put secrets in Git, even base64-encoded.*

**Q3: What is a Kubernetes HPA (HorizontalPodAutoscaler)? Configure one for your AI service that scales based on CPU and Kafka consumer lag. What are the scaling limits to consider?**
*Reason: HPA: automatically scales pod count based on metrics. CPU-based: scale when average CPU > 70% (adds pods). Kafka lag-based: KEDA ScaledObject, adds pods when consumer lag > threshold per replica. Limits to consider: (1) minReplicas ≥ 2 (never scale to 0 for synchronous services). (2) maxReplicas: set a ceiling — don't let autoscaling exhaust the cluster. (3) Stabilization window: prevent flapping (don't scale down immediately after scale-up). (4) Your LLM API has rate limits — scaling from 5 to 50 pods doesn't help if your LLM allows only 100 requests/minute. (5) Cold start time: if a new pod takes 30 seconds to start, it's too slow for sudden spikes — pre-warm with minReplicas.*

**Q4: What is the difference between resource requests and limits in Kubernetes? What happens if you don't set them on an AI service?**
*Reason: requests: the minimum CPU/memory guaranteed to the pod. Used for scheduling (Kubernetes won't place pod on a node with less than this). limits: the maximum allowed. CPU: if exceeded, the pod is throttled (slows down, doesn't crash). Memory: if exceeded, the pod is killed (OOMKilled). Without limits: one AI service pod can consume all node memory → OOMKills all other pods on the same node → cascading failure. Without requests: Kubernetes may over-schedule (more pods than the node can handle) → all pods get throttled → latency spikes. Rule: always set both. For AI services: set memory limit high enough for the LLM response buffers, but not unlimited. CPU: LLM streaming is I/O-bound, not CPU-bound — 500m CPU is usually enough per pod.*

**Q5: A rolling deployment fails halfway through: 3 of 6 pods are running the new version, 3 are running the old version. What options do you have?**
*Reason: Options: (1) Wait and investigate: `kubectl rollout status deployment/payment-ai-service` shows the rollout status. Check new pod logs: `kubectl logs <new-pod>`. (2) Rollback: `kubectl rollout undo deployment/payment-ai-service` — K8s restores the previous version on all pods. (3) Fix and push a new version — the rollout resumes where it left off. Prerequisites that prevent this scenario: (1) Readiness probe — K8s won't route traffic to new pod until it passes. (2) Rollout strategy: `maxUnavailable: 0, maxSurge: 1` — ensures at least 100% of old pods are always running during rollout. (3) CI tests must catch failures before deploying. In production: always test the image with `kubectl run --image=... --rm -it -- /bin/sh` before rolling out.*

**Q6: Explain the difference between AWS RDS (managed PostgreSQL) and self-managed PostgreSQL for a vector database workload. When would you choose each?**
*Reason: RDS: managed → AWS handles backups, failover, patching, scaling storage. Multi-AZ: standby replica in another AZ, automatic failover in 60-120 seconds. Cost: 2x vs single-AZ. Cons: no superuser (can't install arbitrary extensions), less control. For pgvector: pgvector is supported on RDS PostgreSQL 15+. Self-managed PostgreSQL on EC2: full control (any extension, any config), but you manage backups, failover, patching. Use RDS when: you don't have dedicated DBAs, want managed backups and failover, scale is < 50M vectors. Self-managed when: you need custom configurations, very high throughput (> 10K vector searches/second), or cost at very large scale. For most fintech companies: use RDS — the operational burden of self-managed PostgreSQL is not worth it unless you're at very large scale.*

**Q7: What is Terraform and why should infrastructure be treated as code? What are the risks of not using IaC?**
*Reason: Terraform: define infrastructure (VPC, EKS, RDS, Kafka) in .tf files → `terraform apply` creates the real infrastructure. IaC benefits: (1) Reproducible — same Terraform = same infrastructure in dev, staging, prod. (2) Version controlled — infrastructure changes go through code review and Git. (3) Auditable — history of what changed, when, by whom. (4) Disaster recovery — if you lose your prod environment, `terraform apply` recreates it. Without IaC: (1) "Snowflake servers" — each environment is slightly different (works in dev, fails in prod). (2) No audit trail of infrastructure changes. (3) Recovery from disaster is manual and error-prone (6-hour recovery vs 20-minute Terraform apply). (4) Knowledge silos — only the person who clicked in the AWS console knows what was configured. In fintech: infrastructure changes are subject to change management — IaC makes this auditable.*

**Q8: How do you do zero-downtime deployments in Kubernetes? What Spring Boot configuration is needed to support graceful shutdown?**
*Reason: K8s: rolling update + readiness probe → K8s only routes traffic to ready pods → new pod passes readiness → old pod gets SIGTERM → Spring Boot graceful shutdown. Spring Boot graceful shutdown: `server.shutdown: graceful` + `spring.lifecycle.timeout-per-shutdown-phase: 30s`. On SIGTERM: (1) Spring stops accepting new requests. (2) Finishes in-flight requests (up to 30s). (3) Closes database connections cleanly. (4) Process exits. K8s role: sends SIGTERM → waits `terminationGracePeriodSeconds` (default 30s) → sends SIGKILL. For AI services: in-flight LLM calls may take 10+ seconds — set `terminationGracePeriodSeconds: 60` and `spring.lifecycle.timeout-per-shutdown-phase: 45s`. Without graceful shutdown: K8s kills the pod while LLM is generating → customer sees a 500 error mid-streaming.*

**Q9: Design the network security for a Kubernetes-deployed AI payment system. Which services should be accessible from the internet and which should be internal only?**
*Reason: Internet-facing (via Ingress/LoadBalancer): only the API Gateway / BFF (Backend for Frontend). Internal only (ClusterIP): payment-ai-service, fraud-detection-service, qdrant, ollama. Accessible only from specific pods (NetworkPolicy): PostgreSQL/RDS: only from payment-service pods. Kafka: only from producer/consumer service pods. Qdrant: only from AI service pods. Ollama: only from AI service pods. Network Policies: `podSelector: matchLabels: app: qdrant` + `ingress: from: podSelector: matchLabels: app: payment-ai-service`. This restricts Qdrant to only receive traffic from the AI service pod. Internet attack surface: only the ingress controller IP. Even if an attacker compromises one internal service, they can't reach the database directly.*

**Q10: Your AI service is deployed on EKS and costs £8,000/month. Half the cost is EC2 nodes that are 30% utilized. What Kubernetes and AWS strategies would you use to cut costs?**
*Reason: Strategies: (1) Cluster Autoscaler: scale down nodes when utilization is low (but > 30% is already above the default threshold of 50% → check if it's configured). (2) Spot instances: replace on-demand nodes with spot (60-80% cheaper). Add a spot node group alongside on-demand. Stateless pods tolerate interruption. (3) Right-size instances: if using m5.xlarge (4 vCPU, 16GB) and only using 30% of 16GB = 4.8GB → switch to m5.large (2 vCPU, 8GB). (4) KEDA scale-to-zero: async workers (fraud detection consumer) scale to 0 when idle. (5) Schedule non-prod environments to shut down on evenings/weekends (cluster autoscaler → 0 nodes). (6) Graviton instances (ARM): 20% cheaper, same performance for JVM workloads. Target: £8K → £3-4K with spot + right-sizing + scale-to-zero.*
