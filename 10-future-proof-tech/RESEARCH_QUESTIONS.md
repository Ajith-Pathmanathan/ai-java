# Module 10: Future-Proof Technology + Leadership — Research Questions

**Q1: What are Java virtual threads (Project Loom)? How do they change the way you design AI services that make multiple concurrent LLM API calls?**
*Reason: Virtual threads (Java 21): each task gets a lightweight virtual thread (not an OS thread). 1 OS thread can multiplex millions of virtual threads. Blocking I/O (LLM API call) suspends the virtual thread but doesn't block the OS thread. Before: 200-thread pool → max 200 concurrent LLM calls → queue builds up under load. With virtual threads: 10K concurrent LLM calls with the same 8 OS threads. Design impact: you no longer need to worry about thread pool sizing for I/O-bound AI services. `spring.threads.virtual.enabled=true` in Spring Boot 3.2+ — everything uses virtual threads automatically. Still use async/reactive for: CPU-intensive work (embedding inference), true event-driven architectures (WebFlux). Virtual threads are most valuable for: LLM API calls, database queries, external service calls.*

**Q2: What is GraalVM Native Image? What are the benefits and trade-offs for a Spring Boot AI service? When is it worth the complexity?**
*Reason: Native Image: compiles Spring Boot to a platform binary at build time (AOT — Ahead of Time). Benefits: startup in < 100ms (vs 5-10 seconds JVM), 50-80% less memory, smaller container. Trade-offs: 5-10 minute build time (vs 30s for JVM), reflection requires explicit configuration hints, dynamic class loading is restricted. Worth it when: AWS Lambda (cold start penalty eliminates JVM start time advantage), pods that start/stop frequently, memory-constrained environments. Not worth it when: long-running production services (JVM JIT eventually matches native throughput), complex Spring configurations with lots of reflection (Spring Security SAML, etc.). For AI services: most LLM and embedding calls use REST clients → native image compatible. Test your specific Spring AI and LangChain4j configuration — some serialization may need hints.*

**Q3: What is Quarkus? How does it differ from Spring Boot for Kubernetes-native AI applications?**
*Reason: Quarkus: built for Kubernetes and native image from day one. Differences: (1) Native compilation: Quarkus native works out of the box (fewer reflection hints needed than Spring Boot). (2) Dev mode: `quarkus dev` hot-reloads in < 1 second vs Spring DevTools at 3-5 seconds. (3) Reactive: Quarkus Mutiny is the native reactive API (vs Spring WebFlux). (4) Extension ecosystem: similar to Spring starters, includes Quarkus LangChain4j extension. (5) Memory: Quarkus native uses 50MB RAM vs Spring Boot native at 150MB. When to use Quarkus: greenfield microservices, teams targeting native image first, reactive architectures. When to stay with Spring Boot: existing Spring codebase (migration cost), Spring Security SAML/LDAP integrations, teams with deep Spring expertise. 2025: Spring Boot is still 70% of enterprise Java. Know both.*

**Q4: What is structured concurrency (Java 21)? How does it improve over CompletableFuture.allOf() for parallel AI agent calls?**
*Reason: Structured concurrency: a group of related tasks are managed together — if one fails, the others are automatically cancelled. Solves the "lost exception" problem of CompletableFuture: if one future throws, others keep running and you must manually cancel them. Code: `try (var scope = new StructuredTaskScope.ShutdownOnFailure()) { Future<A> a = scope.fork(taskA); Future<B> b = scope.fork(taskB); scope.join().throwIfFailed(); use(a.get(), b.get()); }`. For AI agents: call 3 specialist agents in parallel — if one times out, automatically cancel the others (don't pay for their tokens). CompletableFuture.allOf doesn't cancel on failure — you get the result of all futures even if one failed. Structured concurrency is cleaner and safer for parallel AI calls.*

**Q5: What are reasoning models (OpenAI o3, Gemini Thinking)? When should you use them instead of standard LLMs in a Java AI system?**
*Reason: Reasoning models: the LLM "thinks" before answering — allocates more computation to complex problems. o3: solves math olympiad problems, complex code analysis. 5-10x more expensive and slower (30-120 seconds). Standard LLMs (GPT-4o): fast (1-5s), good for most language tasks. Use reasoning models for: (1) Complex fraud pattern analysis (multiple correlated signals over time). (2) Architectural code review (does this design have correctness issues?). (3) Mathematical financial calculations (compound interest, risk models). (4) When GPT-4o gives wrong answers on a specific task — try a reasoning model. Do NOT use for: classification, extraction, RAG generation, any task where GPT-4o already works. Cost rule: reasoning models are 5-10x more expensive — A/B test before committing. Java integration: same OpenAI API, just specify model name `o3-mini` or `o3`.*

**Q6: What are multimodal AI capabilities and what Java-based use cases do they enable in fintech?**
*Reason: Multimodal: LLM can understand images, audio, video, not just text. GPT-4o Vision, Gemini 1.5 Pro. Java use cases in fintech: (1) Receipt processing: photo of receipt → extract amount, merchant, date (OCR-free). (2) Cheque processing: photo of handwritten cheque → extract amount, account number, signature. (3) KYC document validation: photo of passport/ID → extract name, DOB, verify not expired. (4) Chart analysis: upload a financial chart → LLM interprets trends. Spring AI multimodal: `UserMessage.of("Analyze this receipt", new MediaData(MediaType.IMAGE_JPEG, imageBytes))`. Cost: multimodal tokens are more expensive (images count as many tokens). Use for: document processing where OCR is unreliable (handwriting, poor scan quality). Don't use for: machine-readable documents (use PDF parser instead).*

**Q7: How do you stay current with the rapidly changing AI technology landscape as a Java engineer? What is your filter for which new technologies to learn?**
*Reason: Filter framework: (1) Is there a Java/Spring implementation? (Python-only tools → skip unless you're building the Java equivalent). (2) Does it solve a problem I currently have or will have in 6 months? (Speculative learning wastes time). (3) Is it backed by a major vendor or has > 5K GitHub stars? (Experimental projects often die). (4) Can I validate it in a weekend prototype? (If not, too complex for now). Sources to monitor: Spring AI GitHub releases (weekly), LangChain4j changelog, OpenAI platform changelog, Anthropic changelog, Java Inside Dev newsletter, InfoQ Java topic. Time budget: 1 hour/week current awareness, 4 hours/quarter deep dive on one new technology. Signal vs noise: if 3 independent engineers at different companies mention the same technology → it's worth learning.*

**Q8: What leadership behaviors make a Java engineer an indispensable architect vs just a good developer?**
*Reason: Indispensable architects: (1) Drive decisions, don't just implement them. "We should use pgvector because..." vs "Should I use X or Y?" (2) Write things down. ADRs, design docs, post-mortems. Knowledge doesn't leave with them. (3) Mentor proactively. Teach trade-offs, not solutions. (4) Communicate up AND down. Translate technical constraints for product teams, translate business requirements for engineers. (5) Own outcomes, not just tasks. "The feature works" vs "The feature works AND I validated it in production AND I monitored it for 2 days." (6) Make the team better. Improve processes, code standards, runbooks. Good developers: implement features correctly. Architects: decide WHICH features to build, HOW to build them at scale, and ensure the team can maintain them without the architect being present.*

As more tools become MCP-compatible, the ecosystem creates network effects: one MCP server → works with all AI assistants. Future: MCP marketplaces (AI app stores), enterprise MCP catalogs, MCP security standards. Build MCP servers now while it's differentiating — it will be table stakes by 2026.*
