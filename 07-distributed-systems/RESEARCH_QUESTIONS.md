# Module 07: Distributed Systems + Event-Driven — Research Questions

**Q1: What is the CAP theorem? A fintech startup is choosing between PostgreSQL and Cassandra for their payment ledger. Which would you recommend and why?**
*Reason: CAP: Consistency (every read gets latest write) + Availability (always responds) + Partition Tolerance (survives network splits) — pick two. PostgreSQL = CP: consistent and partition tolerant, will reject writes during network partition to prevent inconsistency. Cassandra = AP: always available, may serve stale data during partition. For a payment ledger: recommend PostgreSQL (CP). Reason: double charges are catastrophic, stale balance reads are unacceptable. A payment system MUST choose consistency over availability — it's better to return an error than to process a payment with stale data. Cassandra is the right choice for: user activity logs, time-series metrics, recommendation data — anything where eventual consistency is acceptable.*

**Q2: What is CQRS? Design the CQRS architecture for a payment system with AI-powered analytics. Include the write side, read side, and how they stay in sync.**
*Reason: CQRS = separate Write model (commands: create payment, update status) from Read model (queries: search payments, get analytics). Write side: `PaymentCommandService` → PostgreSQL (source of truth) → publishes `PaymentCreated`/`PaymentUpdated` events to Kafka. Sync mechanism: Kafka consumer reads events → updates read-model (denormalized, optimized for queries, e.g., Elasticsearch or a PostgreSQL view). AI side: a separate AI-enrichment consumer reads events → runs embeddings → updates vector store. Benefits: read models can be rebuilt by replaying Kafka events; AI analysis doesn't block payment writes; read models optimized per use case (one for customer dashboard, one for fraud analytics).*

**Q3: What is Event Sourcing? How is it different from standard state-based persistence? Give one advantage and one disadvantage for a fintech application.**
*Reason: Standard: store current state (UPDATE payments SET status='COMPLETED'). Event Sourcing: store events (INSERT INTO payment_events: CREATED, AUTHORIZED, CAPTURED). Current state = replay of all events. Advantage: complete audit trail — every state change recorded with who, what, when. Required by financial regulators (FCA, PCI DSS). Time travel: reconstruct payment state at any point. Disadvantage: querying is complex — you need projections (read models) built from events. Without a projection, getting "all completed payments for customer C-789" requires replaying every event. Event Sourcing is powerful but adds complexity — use it when the audit trail and time-travel capabilities justify it (payment systems, trading, compliance).*

**Q4: What is the Saga pattern? Compare choreography vs orchestration sagas. For a payment dispute that involves: fraud check → merchant notification → wait 48h → AI decision → refund or close, which would you use?**
*Reason: Saga: a sequence of local transactions, each with a compensating transaction if something fails. Choreography: services communicate via events, no central coordinator. Good for: simple 2-3 step workflows with loose coupling. Bad for: complex workflows (hard to trace what's happening). Orchestration: central saga orchestrator manages the workflow, calls each step explicitly. Good for: complex multi-step workflows, human-in-the-loop steps. For the dispute saga: use ORCHESTRATION. Reason: (1) It has 5 steps with conditional branching. (2) It involves a 48-hour wait (scheduler). (3) It needs AI integration at step 4. (4) Tracing requires knowing which step the dispute is in — the orchestrator maintains this state. Choreography would make the 48h wait and state tracking much harder.*

**Q5: What is Kafka consumer idempotency? Why is it critical in financial systems? Write a Java implementation.**
*Reason: Kafka guarantees at-least-once delivery — the same event may be delivered twice (during consumer restart, rebalance). In financial systems, processing a payment event twice = double charge. Idempotency = same event processed twice → same result (no duplicate action). Implementation: before processing, check a `processed_events` table for the event ID. If found, skip (already processed). If not found, process and insert the event ID atomically (in the same DB transaction as the business action). SQL: `INSERT INTO processed_events (event_id, processed_at) ON CONFLICT DO NOTHING` — returns 0 rows if already processed, 1 if new. This check must be in the SAME transaction as the payment update to guarantee atomicity.*

**Q6: How do you design for zero-downtime deployment of a service that produces and consumes Kafka events? What happens during a rolling deployment?**
*Reason: Rolling deployment: Pods are upgraded one at a time. During upgrade, old version and new version run simultaneously. Problems: (1) If you change the Kafka event schema (add a field), old consumers can't deserialize new events. (2) If you change the topic name, old producers publish to old topic, new consumers read from new topic — gap in processing. Solutions: (1) Schema evolution with Avro + Schema Registry — backward-compatible changes only. (2) Consumer groups: new version joins the consumer group → Kafka rebalances partitions → traffic shifts gradually. (3) Never rename topics — add new topics and drain old ones. (4) Test with compatibility mode: BACKWARD (new schema can read old data). This is a standard DevOps question at senior level.*

**Q7: What is the Outbox Pattern and why is it essential for reliable event publishing with databases?**
*Reason: Problem: `paymentRepository.save(payment)` + `kafkaTemplate.send(event)` — if the DB saves but Kafka publish fails, the event is lost. The payment exists but nobody knows about it. Outbox Pattern: instead of publishing directly to Kafka, INSERT the event into an `outbox` table in the SAME database transaction as the payment. A background job (Debezium or a poller) reads from the `outbox` table and publishes to Kafka. If the background job fails, it retries. The event is eventually published. Why: this gives you transactional consistency between DB writes and Kafka publishes. Without it, you have a gap where the payment is saved but the event is not published — and no retry mechanism.*

**Q8: Design the Kafka topic strategy for a payment platform. How many topics? How many partitions per topic? What is the message key?**
*Reason: Topics: one per event type (payment-created, payment-completed, payment-failed, fraud-alerts, refund-initiated). Don't put all events in one topic — consumers can subscribe to just what they need. Partitions: partitions = max consumers in the group that can run in parallel. For 10 consumer pods, use 10+ partitions. More partitions = more parallelism = more throughput. Rule of thumb: partitions = max_consumers × 2 (to allow consumer group scaling). Message key: use the payment ID as the key. Kafka guarantees ordering within a partition for the same key. So all events for payment PAY-123 go to the same partition → consumers process them in order (CREATED before COMPLETED). Without a key: events are distributed round-robin → ordering is not guaranteed → consumer might see COMPLETED before CREATED.*

**Q9: What is eventual consistency? How do you handle it in a CQRS system where a user expects to see their just-submitted payment immediately?**
*Reason: Eventual consistency: after a write, the read model will be updated... eventually (usually milliseconds). Problem: user submits payment → gets 200 OK → immediately requests GET /payments/PAY-123 → read model hasn't updated yet → gets 404. Solutions: (1) Read your own writes: after a command, return the result directly from the command service (not the read model). The response contains the payment data. (2) Optimistic update: frontend displays the payment immediately assuming success, confirms when the read model catches up. (3) Async status: return 202 Accepted → client polls GET /payments/PAY-123/status → returns PROCESSING until read model updates. (4) WebSocket push: server pushes an event to the client when the read model updates. Most production systems use option 1 for simple cases, option 3 for complex async operations.*

**Q10: Explain the "Two Generals Problem" and why distributed systems can never guarantee exactly-once delivery across services. How do you design payment systems to handle this?**
*Reason: Two Generals Problem: two armies trying to coordinate an attack via messengers. Messenger might not arrive (network failure). Even with acknowledgments, you can never be 100% sure both sides act. Applied: when Service A calls Service B via HTTP or Kafka, either can fail without knowing the other's state. Guarantee: exactly-once is impossible in distributed systems (proven mathematically). Instead: design for at-least-once + idempotency. Payment system design: (1) HTTP calls → idempotency keys (PUT /payments/{idempotencyKey} — retry-safe). (2) Kafka → idempotent consumers (check processed_events table). (3) Financial transactions → double-entry bookkeeping (every debit has a corresponding credit — inconsistency becomes detectable). (4) Reconciliation jobs → regularly compare payment records between services to detect gaps. This is why fintech has T+1 reconciliation — you check everything the next day.*
